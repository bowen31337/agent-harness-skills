# Harness Plan

Generate a structured **execution plan** from a plain-language description, a
ticket reference, or a feature ID from the claw-forge state service.

The plan is written as a YAML file under `docs/exec-plans/` and optionally
registered with the state service so that agents, `/harness:status`, and the
plan-to-PR traceability convention can track it immediately.

---

## Usage

```bash
# Create a plan from a description (interactive)
/harness:plan "Add refresh-token rotation to the auth service"

# Create from a claw-forge feature ID (fetches context automatically)
/harness:plan FEAT-42

# Create from a ticket reference (used as plan title; description prompted)
/harness:plan JIRA-1234

# Specify the output file path explicitly
/harness:plan "DB migration for user profiles" --output docs/exec-plans/PLAN-005-db-migration.yaml

# Override the auto-assigned plan ID
/harness:plan "Refactor rate limiting" --plan-id PLAN-007

# Set the complexity profile (controls how many task placeholders are generated)
/harness:plan "Logging overhaul" --profile standard
/harness:plan "Security hardening" --profile advanced

# Preview the YAML without writing to disk
/harness:plan "Cache invalidation" --dry-run

# Create the plan then immediately register it with the state service
/harness:plan "Onboarding UI" --register

# Chain into context discovery after plan creation
/harness:plan "Auth service refactor" --then context
```

## Instructions

### Step 1 — Resolve the input

Determine whether the argument is a **feature / ticket ID** or a **free-text description**:

```
FEATURE_ID_PATTERN  = /^[A-Za-z]+-\d+$/      # e.g. FEAT-42, JIRA-1234, PLAN-007
FREE_TEXT            = anything else
```

#### 1A — Feature ID path

Query the state service for the feature's description and tasks:

```bash
FEATURE_ID="<argument>"
STATE_URL="${CLAW_FORGE_STATE_URL:-http://localhost:8888}"

curl -sf "$STATE_URL/features/$FEATURE_ID" 2>/dev/null
```

Extract from the JSON response:

| Field | Used as |
|---|---|
| `title` | Plan title |
| `description` | Plan description and task-generation seed |
| `tasks[]` | Pre-populated task stubs (if present) |
| `domain` | Domain tag in the plan YAML |
| `priority` | Default task priority |

If the state service is unreachable, fall through to **1B** using the feature ID
string as the description seed.

#### 1B — Free-text path

Use the provided description directly as the plan title and seed text.

---

### Step 2 — Assign a plan ID

If `--plan-id` was supplied, use that value directly.

Otherwise, derive the next available plan ID by scanning `docs/exec-plans/`:

```bash
mkdir -p docs/exec-plans

# Find the highest existing PLAN-NNN number
LAST_NUM=$(ls docs/exec-plans/PLAN-*.yaml 2>/dev/null \
  | grep -oE 'PLAN-[0-9]+' \
  | grep -oE '[0-9]+' \
  | sort -n \
  | tail -1)

NEXT_NUM=$(( ${LAST_NUM:-0} + 1 ))
PLAN_ID="PLAN-$(printf '%03d' $NEXT_NUM)"
echo "Assigned plan ID: $PLAN_ID"
```

---

### Step 3 — Choose a complexity profile

| Profile    | Tasks generated | Use for |
|---|---|---|
| `starter`  | 3–5 task stubs  | Small features, bug fixes |
| `standard` | 5–8 task stubs  | Medium features, service changes (default) |
| `advanced` | 8–12 task stubs | Multi-agent epics, cross-cutting concerns |

Default is `standard`.

Derive a **kebab-slug** from the title for file naming and branch suggestions:

```python
import re

title = "<resolved plan title>"
slug = re.sub(r'[^a-z0-9]+', '-', title.lower()).strip('-')[:50]
print("slug:", slug)
```

---

### Step 4 — Generate the plan YAML

Write `docs/exec-plans/${PLAN_ID}-${slug}.yaml` with the following structure.
Expand the `tasks:` list according to the chosen profile and the description
semantics — generate realistic task names that reflect the actual work implied by
the description.

```yaml
# ── Execution Plan ────────────────────────────────────────────────────────────
# Generated by /harness:plan  ·  <TIMESTAMP>
# Edit task titles, priorities, and depends_on freely.
# Register with the state service: PATCH http://localhost:8888/features/<PLAN_ID>
# ─────────────────────────────────────────────────────────────────────────────

plan_id: <PLAN_ID>
title: "<resolved plan title>"
description: |
  <description paragraph — use the description from the state service
   or the free-text input, cleaned up and formatted>
status: pending
priority: medium           # critical | high | medium | low
domain: ""                 # e.g. auth, payments, ui — fill in if known
profile: <starter|standard|advanced>

created_at: "<ISO-8601 UTC timestamp>"
updated_at: "<ISO-8601 UTC timestamp>"
created_by: "harness:plan"

# Traceability (filled in by /review-pr or the plan-to-PR convention)
linked_prs: []

# ── Tasks ─────────────────────────────────────────────────────────────────────
tasks:
  - task_id: TASK-001
    title: "<first logical step>"
    description: ""
    status: pending          # pending | running | done | blocked | skipped
    priority: high           # critical | high | medium | low
    assigned_agent: null
    lock_status: unlocked    # unlocked | locked | done
    depends_on: []
    files_touched: []
    started_at: null
    completed_at: null
    notes: null

  - task_id: TASK-002
    title: "<second logical step>"
    description: ""
    status: pending
    priority: medium
    assigned_agent: null
    lock_status: unlocked
    depends_on:
      - TASK-001
    files_touched: []
    started_at: null
    completed_at: null
    notes: null

  # ... additional tasks per profile
```

**Task generation rules:**

- Task titles must be **imperative verb phrases** ("Scaffold X", "Write tests for Y",
  "Wire middleware Z", "Review and document W").
- `depends_on` must form a **DAG** — no cycles.
- The first task in the list should always have `depends_on: []`.
- For `standard` and `advanced` profiles, include at least one task for tests and
  one for documentation / changelog update.
- Never emit more than 12 tasks; prefer depth (explicit subtasks) over breadth.

---

### Step 5 — Write the file (unless `--dry-run`)

```bash
OUTPUT_FILE="${OUTPUT:-docs/exec-plans/${PLAN_ID}-${slug}.yaml}"

if [ -f "$OUTPUT_FILE" ]; then
  echo "WARNING: $OUTPUT_FILE already exists — aborting to avoid overwrite."
  echo "  Use --plan-id to assign a different ID, or delete the existing file first."
  exit 1
fi

# Write the generated YAML (use Write tool)
echo "Writing: $OUTPUT_FILE"
```

In `--dry-run` mode, print the full YAML to stdout and exit without writing.

---

### Step 6 — Register with the state service (if `--register`)

```bash
STATE_URL="${CLAW_FORGE_STATE_URL:-http://localhost:8888}"

# POST to create the plan entry
curl -sf -X POST "$STATE_URL/features" \
  -H "Content-Type: application/json" \
  -d "{
    \"id\": \"$PLAN_ID\",
    \"title\": \"<plan title>\",
    \"description\": \"<plan description>\",
    \"status\": \"pending\",
    \"priority\": \"medium\"
  }" 2>/dev/null && echo "Registered with state service: $STATE_URL/features/$PLAN_ID" \
  || echo "WARNING: State service unavailable — plan registered locally only."
```

If the state service returns a non-2xx status or is unreachable, warn but do not
fail — the plan YAML on disk is the authoritative source.

---

### Step 7 — Emit the creation summary

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Harness Plan — created
  Plan ID:  <PLAN_ID>  ·  Profile: <profile>
  Title:    <plan title>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  File      : docs/exec-plans/<PLAN_ID>-<slug>.yaml
  Tasks     : <N> tasks  (<pending> pending · 0 running · 0 done)
  State SVC : <registered | local only>

  Suggested branch
  └─ feat/<PLAN_ID>-<slug>

  Next steps
  ├─ Review and adjust task list in the YAML file.
  ├─ Run /harness:context <PLAN_ID> to discover relevant files.
  ├─ Run /harness:status to see the plan in the dashboard.
  └─ Commit: git add docs/exec-plans/<PLAN_ID>-<slug>.yaml

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

In `--dry-run` mode, prefix the header with `[DRY-RUN]` and replace "File" with
"Would write".

---

### Step 8 — Handle `--then` chaining (if requested)

If `--then context` is passed, immediately invoke `/harness:context <PLAN_ID>` to
discover the relevant files for the new plan and present the context manifest.

Other supported chain targets:

| `--then` value | Effect |
|---|---|
| `context` | Run `/harness:context <PLAN_ID>` |
| `status` | Run `/harness:status --plan-file <OUTPUT_FILE>` |
| `lint` | Run `/harness:lint` against the new plan file |

When operating as an agent (not using the CLI directly), invoke each chained step
as a separate tool call and check the result before proceeding.

## Options

| Flag | Default | Effect |
|---|---|---|
| `--plan-id PLAN_ID` | auto-assigned | Override the generated `PLAN-NNN` identifier |
| `--output PATH` | `docs/exec-plans/<id>-<slug>.yaml` | Destination path for the plan file |
| `--profile PROFILE` | `standard` | Task complexity profile: `starter`, `standard`, or `advanced` |
| `--register` | off | POST the plan to the claw-forge state service after writing |
| `--dry-run` | off | Print the plan YAML to stdout; do not write to disk |
| `--then SUBCMD` | — | Chain a harness subcommand after plan creation (e.g. `--then context`) |
| `--state-url URL` | `http://localhost:8888` | Override the state service base URL |

Environment:
- `CLAW_FORGE_STATE_URL` — overrides `--state-url` default

## Output schema

The generated YAML conforms to `harness_skills.models.plan.ExecutionPlan`.

Import in Python:

```python
from harness_skills.models.plan import ExecutionPlan
import yaml, pathlib

raw = pathlib.Path("docs/exec-plans/PLAN-001-auth-refresh-token.yaml").read_text()
plan = ExecutionPlan.model_validate(yaml.safe_load(raw))
print(plan.plan_id, plan.status, len(plan.tasks))
```

Key model fields:

| Field | Type | Description |
|---|---|---|
| `plan_id` | `str` | Unique plan identifier (e.g. `PLAN-001`) |
| `title` | `str` | Human-readable plan title |
| `description` | `str` | Full description of the work |
| `status` | `PlanStatus` | `pending \| running \| done \| blocked \| cancelled` |
| `priority` | `Priority` | `critical \| high \| medium \| low` |
| `domain` | `str \| None` | Service / domain tag |
| `profile` | `str` | Complexity profile used to generate the task list |
| `created_at` | `datetime` | ISO-8601 UTC creation timestamp |
| `updated_at` | `datetime` | ISO-8601 UTC last-update timestamp |
| `created_by` | `str` | Agent or user that created the plan |
| `linked_prs` | `list[LinkedPR]` | PRs opened to deliver this plan |
| `tasks` | `list[Task]` | Ordered task list (see below) |

`Task` fields:

| Field | Type | Description |
|---|---|---|
| `task_id` | `str` | Unique task identifier (e.g. `TASK-001`) |
| `title` | `str` | Imperative-verb task title |
| `description` | `str \| None` | Optional expanded description |
| `status` | `TaskStatus` | `pending \| running \| done \| blocked \| skipped` |
| `priority` | `Priority` | Task-level priority |
| `assigned_agent` | `str \| None` | Agent currently responsible |
| `lock_status` | `LockStatus` | `unlocked \| locked \| done` |
| `depends_on` | `list[str]` | `task_id`s this task depends on (DAG edges) |
| `files_touched` | `list[str]` | Paths modified by this task (filled in at execution time) |
| `started_at` | `datetime \| None` | When the task began |
| `completed_at` | `datetime \| None` | When the task finished |
| `notes` | `str \| None` | Free-form notes added during execution |

## Plan-to-PR traceability

Plans created by `/harness:plan` participate in the full plan-to-PR convention
(see `docs/plan-to-pr-convention.md`):

- **Branch naming** — `feat/<PLAN_ID>-<slug>` (suggested in the summary card)
- **PR title** — `[<PLAN_ID>] <description>`
- **Commit trailer** — `Plan: <PLAN_ID>`
- **Write-back** — after `gh pr create`, update `linked_prs` in the plan YAML

## Notes

- **Never overwrites** — if the target YAML file already exists, the skill aborts
  with a clear error rather than silently clobbering manual edits.
- **Idempotent ID assignment** — plan IDs are derived from the filesystem scan at
  creation time; running the skill twice produces consecutive IDs, not duplicates.
- **State service is optional** — if unreachable, the plan is written to disk and
  the `--register` step is silently skipped with a warning.
- **DAG validation** — the generated `depends_on` graph is always acyclic; the skill
  never creates circular dependencies.
- **Does not auto-commit** — review the generated YAML before committing;
  use `/checkpoint` to snapshot if needed.
- **Pipeline-composable** — combine with context discovery for a zero-friction
  planning-to-execution flow:
  ```bash
  /harness:plan "Auth service refactor" --register --then context
  ```
