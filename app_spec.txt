<project_specification mode="greenfield">
  <project_name>harness-skills</project_name>

  <overview>
    Harness Skills Generator is a comprehensive toolkit that analyzes codebases and generates
    agent harness infrastructure — AGENTS.md files, architectural constraints, golden principles,
    evaluation gates, observability configs, and execution plan frameworks. It shifts the engineer's
    role from writing code to designing environments where AI coding agents (Claude Code, Codex,
    Cursor, etc.) produce reliable, architecturally coherent output at scale. The tool is built as
    a plugin-based Python CLI with a Claude Code skill interface layer.
  </overview>

  <target_audience>
    Engineers and teams using AI coding agents who want structured harnesses for their projects.
    This includes individual developers setting up agent workflows, tech leads defining architectural
    guardrails for agent-assisted teams, and platform engineers building standardized agent
    environments across organizations. Users range from those new to agent-first development
    (who need sensible defaults) to experienced harness engineers (who need customization and
    plugin extensibility).
  </target_audience>

  <core_features>
    <category name="Codebase Analysis &amp; Detection">
      - Skill detects primary language(s) and framework(s) from package files (package.json, pyproject.toml, Cargo.toml, go.mod, Gemfile, pom.xml, etc.)
      - Skill detects project structure pattern (monorepo, polyrepo, single-app) from directory layout and workspace configs
      - Skill identifies existing test framework and test location conventions from config files and test directory structure
      - Skill identifies existing linter/formatter configuration (ESLint, Ruff, rustfmt, Prettier, etc.) and extracts active rules
      - Skill detects CI/CD platform from config files (.github/workflows, .gitlab-ci.yml, Jenkinsfile, .circleci/config.yml)
      - Skill maps directory tree into candidate domain boundaries (e.g., src/auth, src/billing become domains) using heuristics and import clustering
      - Skill detects existing dependency layering patterns from import graph analysis via tree-sitter AST parsing
      - Skill identifies existing documentation files (README, CONTRIBUTING, existing AGENTS.md) and their coverage completeness
      - Skill detects database technology from config files, migration directories, ORM model definitions, and connection strings
      - Skill identifies API style (REST, GraphQL, gRPC) from route definitions, schema files, or proto files
      - Skill detects environment variable patterns from .env.example, config files, and os.environ/process.env references in code
      - Skill produces a harness_manifest.json summarizing all detected context (stack, domains, patterns, conventions) for downstream generators
    </category>

    <category name="AGENTS.md Generation">
      - Skill generates a root AGENTS.md with progressive disclosure structure (overview pointing to deeper per-domain docs)
      - Skill includes detected build, test, and lint commands in the root AGENTS.md with exact invocation syntax
      - Skill generates per-domain AGENTS.md files for each detected subsystem (e.g., src/auth/AGENTS.md, src/billing/AGENTS.md)
      - Each domain AGENTS.md includes: purpose statement, key files listing, internal patterns, domain-specific constraints
      - Skill includes architecture overview section mapping domain boundaries and dependency flow direction
      - Skill includes security protocols section covering secret handling, input validation patterns, and auth conventions
      - Skill includes git workflow section with branch naming conventions, commit message format, and PR process
      - Skill includes code conventions section derived from existing linter config and detected patterns in the codebase
      - Skill includes testing conventions section with test file naming, assertion style, coverage expectations, and fixture patterns
      - Skill includes error handling patterns section derived from existing codebase error handling conventions
      - Skill cross-links all generated documentation so agents can navigate between related files via relative paths
      - Skill enforces token-budget awareness with each AGENTS.md staying under a configurable max token count to avoid context pollution
      - Skill flags stale documentation risks by tracking which source files each AGENTS.md references and detecting drift
      - Skill supports regeneration mode that re-scans codebase and updates existing AGENTS.md files without overwriting manual edits via three-way merge strategy
    </category>

    <category name="Architecture Documentation &amp; Enforcement">
      - Skill generates ARCHITECTURE.md with a domain map showing all detected subsystems and their relationships
      - Skill infers dependency direction between domains from import analysis and generates a text-based dependency flow diagram
      - Skill generates a layered architecture definition per domain (Types to Config to Repo to Service to Runtime to UI or project-appropriate equivalent)
      - Skill detects violations of the inferred dependency layers in existing code and reports them as warnings with file locations
      - Skill generates linter rules that mechanically enforce dependency direction (ESLint import rules, Ruff import checks, Go import restrictions, etc.)
      - Skill generates CI validation script that fails builds on architectural boundary violations
      - Skill generates a providers pattern definition for cross-cutting concerns (logging, auth, config) that domains must use instead of direct imports
      - Skill documents the one-way dependency rule: each layer may only import from layers below it, never above or sideways
      - Skill generates file naming convention rules and a linter config to enforce them based on detected conventions
      - Skill generates file size limit rules (configurable threshold) to prevent monolithic files that confuse agents
      - Skill generates module boundary rules where public API surface per domain is explicitly declared in index/init files
      - Skill generates docs/design-docs/ directory structure with a template for architectural decision records (ADRs)
      - Skill generates docs/generated/ directory for auto-derived documentation (database schemas, API specs, dependency graphs)
      - Skill supports custom layer definitions allowing engineers to override the default layer stack for their architecture style
      - Skill generates a structural test suite that validates all architectural invariants are maintained
      - Skill generates a harness lint command (shell script or Makefile target) that runs all architectural checks in one pass
    </category>

    <category name="Golden Principles &amp; Pattern Enforcement">
      - Skill analyzes the codebase for recurring code patterns and extracts them as candidate golden principles with frequency data
      - Skill generates a PRINCIPLES.md file listing all mechanical rules agents must follow with rationale and examples
      - Skill generates a prefer-shared-utilities-over-hand-rolled-helpers rule with pointers to existing utility packages in the codebase
      - Skill generates boundary-level validation rules specifying that data validation happens at system boundaries (API input, external services) not deep in business logic
      - Skill generates concurrency and async patterns rule based on detected framework conventions (async/await patterns, thread safety rules, pool usage)
      - Skill generates error handling pattern rules specifying structured errors, error codes, and logging format conventions
      - Skill generates naming convention rules for functions, variables, files, and database columns derived from existing codebase style analysis
      - Skill generates import ordering and grouping rules matching existing conventions detected from the most common patterns
      - Skill generates a no-magic-numbers and no-hardcoded-strings rule with config pattern guidance and constant extraction conventions
      - Skill generates test writing principles covering arrange-act-assert structure, test naming conventions, fixture patterns, and mock boundary rules
      - Skill supports custom principle definitions allowing engineers to add project-specific golden rules via interactive prompt or config file
      - Skill generates background cleanup task definitions that agents can run to enforce principles across the entire codebase as refactoring PRs
      - Skill generates a principle violation report command that scans the codebase and lists all deviations from golden principles with severity and location
    </category>

    <category name="Evaluation Gates &amp; Testing Harness">
      - Skill generates an EVALUATION.md defining all completion criteria agents must satisfy before opening a PR
      - Skill generates a regression test gate ensuring all existing tests must pass before any change is accepted
      - Skill generates a coverage gate with configurable threshold (default 90%) that blocks PRs falling below the bar
      - Skill generates a security check gate covering secret scanning, dependency vulnerability audit, and input validation verification
      - Skill generates a performance benchmark gate with configurable thresholds (e.g., no API endpoint exceeds 500ms, no span exceeds 2 seconds)
      - Skill generates an architectural compliance gate that runs the structural test suite from the architecture enforcement component
      - Skill generates a golden principles compliance gate that runs the principle violation scanner and fails on critical violations
      - Skill generates a documentation freshness gate that verifies AGENTS.md files reference files that still exist and content is current
      - Skill generates a type safety gate that runs the type checker (TypeScript strict, mypy, etc.) with zero errors required
      - Skill generates a lint gate that runs all configured linters with zero warnings required on changed files
      - Skill generates a harness evaluate command that runs all gates in sequence and produces a structured pass/fail report
      - Skill generates per-gate configuration in harness.config.yaml so engineers can enable, disable, or adjust thresholds per gate
      - Skill generates CI pipeline integration as GitHub Actions workflow and GitLab CI job that runs harness evaluate on every PR
      - Skill generates a gate failure report format with actionable fix suggestions that agents can consume and act on automatically
      - Skill supports custom evaluation gates allowing engineers to define project-specific gates via a plugin interface in harness.config.yaml
    </category>

    <category name="Observability &amp; Application Legibility">
      - Skill generates structured logging configuration matching the detected framework (Python logging, Winston, slog, tracing, etc.)
      - Skill generates a logging convention document specifying required fields per log entry (timestamp, level, domain, trace_id, message)
      - Skill generates log format linter rules ensuring all log statements follow the structured pattern
      - Skill generates a per-worktree boot script that launches an isolated application instance for each agent task
      - Skill generates environment isolation configuration with separate ports, database schemas, or containers per worktree
      - Skill generates a health check endpoint specification that agents can poll to verify the application is running correctly
      - Skill generates a harness boot command that starts the isolated app instance and waits for health check to pass before returning
      - Skill generates a harness observe command that tails structured logs filtered by domain or trace_id in real time
      - Skill generates browser automation integration config (Playwright or Puppeteer) so agents can drive the UI and capture screenshots
      - Skill generates a DOM snapshot utility that agents can invoke to inspect current UI state without a full browser session
      - Skill generates performance measurement hooks so agents can query response times, memory usage, and startup duration
      - Skill generates an error aggregation view so agents can query recent errors grouped by domain and frequency
      - Skill generates a harness screenshot command that captures the current application state as a visual artifact for PR evidence
      - Skill generates observability stack templates (optional) for lightweight log aggregation in local dev using file-based collectors
    </category>

    <category name="Execution Plans &amp; State Management">
      - Skill generates docs/exec-plans/ directory structure with templates for execution plan artifacts
      - Skill generates execution plan template with sections: objective, approach, steps, progress log, known debt, completion criteria
      - Skill generates a harness plan command that creates a new execution plan from a feature description or ticket
      - Skill generates progress log format where agents append timestamped entries as they complete steps within a plan
      - Skill generates a plan status dashboard command (harness status) showing all active, completed, and blocked plans
      - Skill generates context handoff protocol where ending agent sessions write a structured summary to the plan progress log for the next session
      - Skill generates a harness resume command that loads the most recent plan state and presents it as agent context
      - Skill generates git-based checkpoint integration where agents commit work-in-progress to a branch with plan reference in the commit message
      - Skill generates a technical debt tracker where agents log known shortcuts or TODOs into docs/exec-plans/debt.md with severity and remediation notes
      - Skill generates a plan-to-PR linking convention where each PR references its source execution plan for full traceability
      - Skill generates a stale plan detector that flags execution plans with no progress updates beyond a configurable threshold
      - Skill generates a plan completion report summarizing what was done, what debt was incurred, and what follow-up is needed
    </category>

    <category name="Skill Invocation &amp; CLI Interface">
      - Skill registers as /harness:create for full harness generation from codebase analysis through all artifact output
      - Skill registers as /harness:update for re-scanning codebase and updating existing harness artifacts while preserving manual edits
      - Skill registers as /harness:lint for running all architectural and golden principle checks in one pass
      - Skill registers as /harness:evaluate for running all evaluation gates and producing a structured pass/fail report
      - Skill registers as /harness:boot for launching an isolated application instance for an agent task
      - Skill registers as /harness:plan for creating a new execution plan from a description or ticket reference
      - Skill registers as /harness:status for showing current state of all plans, gates, and harness health metrics
      - Skill registers as /harness:resume for loading most recent plan state and presenting it for context handoff
      - Skill registers as /harness:screenshot for capturing application state as a visual artifact
      - Skill generates a harness.config.yaml as the central configuration file for all thresholds, toggles, custom rules, and plugin settings
      - Skill generates a harness init shell script for teams not using Claude Code providing the same functionality via standalone CLI
    </category>
  </core_features>

  <key_interactions>
    <interaction number="1" name="First-time harness creation">
      1. Engineer runs harness create (or /harness:create in Claude Code) in their project root
      2. Scanner analyzes codebase: detects language, framework, domains, patterns, existing docs
      3. Skill presents harness_manifest.json summary and asks for confirmation/adjustments
      4. Engineer confirms or adjusts domain boundaries and layer definitions
      5. Skill generates all harness artifacts: AGENTS.md files, ARCHITECTURE.md, PRINCIPLES.md, EVALUATION.md, configs
      6. Skill generates CI pipeline with evaluation gates
      7. Skill generates harness.config.yaml with detected defaults
      8. Engineer reviews generated artifacts, commits to repository
    </interaction>
    <interaction number="2" name="Harness update after codebase changes">
      1. Engineer adds new modules or refactors existing code
      2. Engineer runs harness update (or /harness:update)
      3. Scanner re-analyzes codebase and detects changes (new domains, moved files, new patterns)
      4. Skill uses three-way merge to update AGENTS.md and other artifacts without overwriting manual edits
      5. Skill reports what changed: new domains discovered, stale references fixed, new patterns detected
      6. Engineer reviews diff and commits
    </interaction>
    <interaction number="3" name="Agent runs evaluation before PR">
      1. Agent completes a coding task in an isolated worktree
      2. Agent runs harness evaluate
      3. System runs all gates in sequence: regression tests, coverage, security, performance, architecture, principles, docs freshness, types, lint
      4. System produces structured report with pass/fail per gate and actionable fix suggestions
      5. If any gate fails, agent consumes fix suggestions and iterates
      6. Once all gates pass, agent opens PR with evaluation report attached
    </interaction>
    <interaction number="4" name="Execution plan lifecycle">
      1. Engineer or agent runs harness plan with a feature description
      2. System creates execution plan in docs/exec-plans/ with steps, criteria, and empty progress log
      3. Agent begins work, appending progress entries as steps complete
      4. If agent session ends mid-task, it writes context handoff summary to progress log
      5. New agent session runs harness resume, loads plan state, continues from last checkpoint
      6. On completion, system generates plan completion report with debt inventory
    </interaction>
    <interaction number="5" name="Isolated application boot for agent task">
      1. Agent receives a bug fix or feature task
      2. Agent runs harness boot to start isolated application instance
      3. System assigns unique port/database, boots app, waits for health check
      4. Agent uses harness observe to tail logs, harness screenshot to capture UI state
      5. Agent drives browser via Playwright to reproduce bug or verify feature
      6. Agent fixes code, re-runs harness evaluate, confirms fix via UI observation
      7. Agent runs harness screenshot for PR evidence
    </interaction>
  </key_interactions>
