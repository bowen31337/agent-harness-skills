# Plan-to-PR Linking Convention

> Full traceability from execution plan to merged pull request.

---

## Why this convention exists

When multiple agents execute tasks in parallel, it becomes difficult to answer:
- "Which PR delivered task TASK-007?"
- "What plan authorised this change?"
- "Are all tasks from PLAN-003 shipped?"

This convention makes those questions answerable in O(1) time by embedding
plan metadata in every PR and writing PR URLs back into every plan file.

---

## Convention rules (summary)

| Artefact     | Required field / pattern |
|--------------|--------------------------|
| Branch name  | `feat/PLAN-NNN-<kebab-slug>` |
| PR title     | `[PLAN-NNN] <short description>` |
| PR body      | Traceability table (see template) |
| Commit body  | `Plan: PLAN-NNN` trailer |
| Plan YAML    | `linked_prs` list updated after PR open |

---

## 1. Branch naming

```
feat/PLAN-NNN-<kebab-slug-of-title>
```

Examples:
- `feat/PLAN-001-auth-refresh-token`
- `feat/PLAN-007-logging-structured-output`

The `PLAN-NNN` prefix lets reviewers and CI identify the source plan without
opening the PR body.

---

## 2. PR title

```
[PLAN-NNN] <imperative short description>
```

Example:
```
[PLAN-001] Add refresh-token rotation to auth service
```

The bracket prefix is machine-parseable. `gh pr list --search "[PLAN-001]"`
returns all PRs for a plan instantly.

---

## 3. PR body — traceability table

Every PR body generated by an agent MUST include the traceability block from
`.github/pull_request_template.md`.  The mandatory section is:

```markdown
## Execution Plan

| Field        | Value |
|--------------|-------|
| Plan ID      | `PLAN-001` |
| Plan file    | `docs/exec-plans/PLAN-001-auth-refresh-token.yaml` |
| Tasks closed | TASK-003, TASK-004 |
| Plan status  | running |
```

If the PR was not generated from an execution plan, the agent MUST write:

```markdown
## Execution Plan

No execution plan — ad-hoc change.
```

Ad-hoc PRs are permitted but must be declared explicitly so they are not
silently missing from the traceability record.

---

## 4. Commit message trailer

Every commit that belongs to a plan must include a `Plan:` trailer in the
commit body so `git log` is independently searchable:

```
feat: rotate refresh tokens on each use

Implements TASK-003 and TASK-004 from the auth hardening plan.

Plan: PLAN-001
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

Agents using the HEREDOC commit pattern (PRINCIPLES.md §4.5) should append
the `Plan:` trailer before `Co-Authored-By:`.

---

## 5. Writing the PR URL back to the plan file

Immediately after `gh pr create` returns the PR URL, the agent MUST update
the plan YAML's `linked_prs` list.

### Procedure

1. Read the current plan file.
2. Append to `linked_prs`:
   ```yaml
   - pr_url: "https://github.com/org/repo/pull/42"
     pr_number: 42
     opened_at: "2026-03-20T14:00:00Z"
     opened_by: "coding-03abe8fb"
     tasks_addressed:
       - TASK-003
       - TASK-004
   ```
3. Update `plan.updated_at` to the current timestamp.
4. Commit the updated plan file with message:
   ```
   chore: link PR #42 to PLAN-001

   Plan: PLAN-001
   ```

The write-back commit MUST be on the feature branch, NOT on `main`.  It
becomes part of the PR diff so reviewers can verify traceability is complete.

---

## 6. Verification checklist (for agents)

Before marking a coding task `done`, verify:

- [ ] Branch follows `feat/PLAN-NNN-<slug>` naming
- [ ] PR title includes `[PLAN-NNN]`
- [ ] PR body traceability table is filled in
- [ ] Commit body includes `Plan: PLAN-NNN` trailer
- [ ] Plan YAML `linked_prs` updated with PR URL
- [ ] Addressed tasks set to `lock_status: done` in plan YAML
- [ ] Plan YAML `updated_at` timestamp refreshed

---

## 7. Querying traceability

### Find all PRs for a plan
```bash
gh pr list --search "[PLAN-001]" --json number,title,url,state
```

### Find the plan for a PR
```bash
# From the PR body
gh pr view 42 --json body | jq '.body' | grep "Plan ID"

# From git log on the branch
git log --grep="Plan: PLAN-" --format="%H %s" feature-branch
```

### Find all plans with open PRs
```bash
grep -r "pr_url" docs/exec-plans/ | grep -v "Example"
```

### Verify all plan tasks have a linked PR
```bash
python skills/exec_plan.py verify-prs PLAN-001
```

---

## 8. Convention enforcement

The `harness-evaluate` CI workflow checks:

1. PR title matches `^\[PLAN-\d{3}\]` (or contains "No execution plan" in body).
2. Plan file exists at the path declared in the traceability table.
3. Plan file `linked_prs` contains an entry referencing this PR number.

PRs that fail these checks are labelled `missing-plan-link` and require a
human override to merge.
